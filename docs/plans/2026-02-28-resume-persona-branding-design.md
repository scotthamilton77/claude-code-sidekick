# Resume Message Persona Branding

**Issue**: sidekick-60t5
**Date**: 2026-02-28

## Problem

Resume messages are generated by whatever persona was active in the prior session. When a user resumes with a different persona, the injected message sounds "off" — wrong voice, wrong tone.

## Solution: Source Persona Branding with Attribution Wrapper

### Schema Change

Add two nullable fields to `ResumeMessageStateSchema`:

- `persona_id: string | null` — source persona ID (null when persona was disabled)
- `persona_display_name: string | null` — source persona's display name for attribution

### Generation (update-summary.ts)

When writing `resume-message.json`, populate `persona_id` and `persona_display_name` from the session's active persona. Both null when persona is disabled.

### Display (statusline-service.ts)

When discovering a previous session's resume message, compare source `persona_id` with current session's `persona_id`:

| Source Persona | Current Persona | Behavior |
|---------------|----------------|----------|
| Same as current | — | Show as-is (current behavior) |
| Different (not null) | Any | `"[display_name]: [snarky_comment]"` |
| null (disabled) | Any | Show as-is |
| Any | null (disabled) | Show as-is |

### Why Attribution Over Regeneration

- No extra LLM calls at resume time
- Original persona's voice is preserved faithfully
- Simple implementation — formatting only, no new LLM plumbing
- Resilient — display_name stored in JSON, no persona YAML lookup needed

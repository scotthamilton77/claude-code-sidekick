# 9.2.3 Reminder Relationship Mapping

## Summary
- Reminder types: 3 (UserPromptSubmit, PauseAndReflect, VerifyCompletion)
- Cross-reminder rules: 4 major rules
- Implementation: Scattered across handlers (no central orchestrator)

## Reminder Type Inventory

| Type | Purpose | Staged By | Consumed By | Priority | Persistent | Blocking |
|------|---------|-----------|-------------|----------|------------|----------|
| user-prompt-submit | Inject CLAUDE.md context on every turn | SessionStart, BulkProcessingComplete | UserPromptSubmit hook | 10 | YES | NO |
| pause-and-reflect | Checkpoint when tools exceed threshold | ToolCall event (transcript) | PreToolUse hook | 80 | NO | YES |
| verify-completion | Force verification before task completion | ToolCall event (source file edit) | Stop hook | 50 | NO | YES |

## Cross-Reminder Rules

### Rule 1: P&R Unstages VC (Cascade Prevention)
- **Trigger**: `StagePauseAndReflect` handler on ToolCall event
- **Effect**: Deletes `Stop/verify-completion.json` reminder
- **Implemented in**: `feature-reminders/handlers/staging/stage-pause-and-reflect.ts:72-75`
- **Rationale**: When P&R blocks the model, Stop hook would fire immediately and consume VC, defeating P&R's purpose. Unstaging VC prevents this cascade.

### Rule 2: UserPromptSubmit Unstages VC (Task Complete)
- **Trigger**: UserPromptSubmit hook event
- **Effect**: Either deletes `Stop/verify-completion.json` OR re-stages it if unverified changes exist
- **Implemented in**: `feature-reminders/handlers/staging/unstage-verify-completion.ts:46-124`
- **Rationale**: New user prompt indicates previous task complete; verification no longer needed. Exception: if LLM marked changes as "unverified", VC re-stages for next Stop.

### Rule 3: VC Consumption Resets P&R Baseline
- **Trigger**: Stop hook consuming verify-completion reminder
- **Effect**: Sends `reminder.consumed` IPC to Daemon, which writes `pr-baseline.json`
- **Implemented in**: `feature-reminders/handlers/consumption/inject-stop.ts:115-139`
- **Rationale**: P&R threshold adjusts relative to where VC was consumed. Without this reset, P&R countdown wouldn't reset and model could work indefinitely without another P&R prompt.

### Rule 4: VC Consumption Unstages P&R (Prevent Double Block)
- **Trigger**: Stop hook consuming verify-completion reminder
- **Effect**: Deletes `PreToolUse/pause-and-reflect.json` via `onConsume` callback
- **Implemented in**: `feature-reminders/handlers/consumption/inject-stop.ts:115-119`
- **Rationale**: Prevents both VC and P&R blocking simultaneously, which would be redundant.

## Counter/Countdown Mechanics

| Counter | Purpose | Reset Trigger | Owner | Location |
|---------|---------|---------------|-------|----------|
| `toolsThisTurn` | Tools since last UserPromptSubmit | New UserPromptSubmit event | TranscriptService | @sidekick/core |
| `turnCount` | Total user prompts in session | Read-only, incremented on UserPromptSubmit | TranscriptService | @sidekick/core |
| `toolCount` | Total tools in session | Read-only, incremented on ToolCall | TranscriptService | @sidekick/core |
| P&R baseline | Effective P&R trigger point | UserPromptSubmit or VC consumed | File: `pr-baseline.json` | stage-pause-and-reflect.ts |
| VC cycle count | Max times VC can re-stage per prompt | Cycle limit reached or new prompt | File: `vc-unverified.json` | unstage-verify-completion.ts |

**Key insight**: No reminders maintain their own counters. All counters derive from TranscriptService (read-only) or are stored as state files for cross-handler coordination.

## Suppression Logic

| Marker | Effect | Created By | Consumed By |
|--------|--------|------------|-------------|
| `pr-baseline.json` | Adjusts P&R threshold baseline | Stop hook consuming VC (via IPC) | P&R staging handler |
| `vc-unverified.json` | Marks changes as unverified for re-staging | Stop hook (non-blocking classification) | UserPromptSubmit handler |
| Timestamp suffix (e.g., `verify-completion.1234567890.json`) | Marks reminder as consumed | CLIStagingReader on consumption | Staging handlers (reactivation check) |

**Reminder States**:
1. **Staged**: Active `.json` file → consumed on next hook
2. **Unstaged**: Deleted via `deleteReminder()` → no effect
3. **Consumed**: Timestamped file → tracked for reactivation decisions

## Priority System

When multiple reminders staged for same hook, highest-priority consumed first:

| Reminder | Priority | Effect |
|----------|----------|--------|
| pause-and-reflect | 80 | Blocks at PreToolUse (highest) |
| verify-completion | 50 | Blocks at Stop (medium) |
| user-prompt-submit | 10 | Informational at UserPromptSubmit (lowest) |

Selection in `CLIStagingReader.listReminders()`: `reminders.sort((a, b) => b.priority - a.priority)`

Only first (highest-priority) reminder consumed per hook invocation.

## Recommendations for 9.6 (Reminder Orchestration)

### What ReminderOrchestrator Should Own

1. **Baseline State Management**
   - Currently: P&R handler reads `pr-baseline.json`, VC writes via IPC
   - Orchestrator: Centralize baseline state service with clear read/write semantics

2. **Cross-Reminder Unstaging Rules**
   - Currently: Scattered `deleteReminder()` calls in 3 handlers
   - Orchestrator: Define as declarative rules:
     ```typescript
     interface ReminderRule {
       trigger: 'pause-and-reflect-staged' | 'user-prompt-submitted' | 'vc-consumed'
       effects: Array<{ action: 'unstage' | 'restage'; reminder: ReminderId }>
     }
     ```

3. **Reactivation Decisioning**
   - Currently: Each handler implements own logic (P&R reads history, VC checks cycle)
   - Orchestrator: Provide `shouldReactivate(reminderName, context)` interface

4. **Smart Completion Classification**
   - Currently: Tightly coupled in inject-stop.ts
   - Orchestrator: Clean interface for classification results and unverified state

### Centralization Priority

**High** (straightforward):
- P&R baseline state (file-based → service-based)
- Unstaging rules (P&R↔VC dependencies)
- Reactivation decisions (table-driven)

**Medium** (more refactoring):
- Smart completion classification (extract from inject-stop.ts)
- Unverified state management (manual file I/O → service)

**Low** (already well-encapsulated):
- Handler registration (modular via index.ts)
- Reminder resolution/templating (reminder-utils.ts)
- CLI staging reader (isolated)

### Proposed Pattern: Rule Engine

```typescript
interface ReminderEvent {
  kind: 'staged' | 'consumed' | 'unstaged'
  reminder: ReminderId
  hook: HookName
  context: { turnCount; toolsThisTurn; ... }
}

interface ReminderRule {
  name: string
  trigger: (event: ReminderEvent) => boolean
  effects: (ctx: RuntimeContext) => Promise<void>
}

const reminderRules: ReminderRule[] = [
  {
    name: 'P&R unstages VC cascade',
    trigger: (e) => e.kind === 'staged' && e.reminder === 'pause-and-reflect',
    effects: async (ctx) => {
      await ctx.staging.deleteReminder('Stop', 'verify-completion')
    }
  },
  // ... more rules
]
```

This eliminates scattered `deleteReminder()` calls and makes cross-reminder logic explicit and testable.

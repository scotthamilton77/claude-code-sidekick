### Coupling Issues

| Issue | Files Involved | Impact |
|-------|----------------|--------|
| Implicit IPC protocol between features and daemon | daemon.ts (handler), feature-reminders (caller) | No formal contract, embedded in switch statement |
| Statusline reads daemon-written files without interface | StatuslineService, daemon handlers | Implicit JSON schema contracts |

| Daemon has reminder-specific knowledge | daemon.ts:625 hardcoded check | Daemon should not know reminder semantics |

## Severity Assessment

### High

1. **Feature code in daemon** (4 items)
   - P&R baseline logic in daemon.ts:624 (has FIXME comment acknowledging this)
   - VC state management (vc-unverified.json, IPC handlers)
   - Reminder name hardcoding in daemon core
   - Impact: Adding new reminder requires modifying daemon

2. **Missing StateWriter abstraction**
   - 40+ write locations with inconsistent patterns
   - No formal contract for state persistence
   - Impact: Statusline indirectly depends on daemon implementation details

3. **Missing PathResolver service**
   - 90+ path constructions across packages
   - Single point of change impossible
   - Impact: Directory structure changes require touching every package

### Medium

1. **Hook name mappings in CLI**
   - Should be in @sidekick/types for sharing
   - Impact: Validation logic duplicated

2. **Hardcoded timeouts and thresholds**
   - Context metrics capture timings
   - Should be configurable
   - Impact: Can't tune without code changes

3. **Implicit IPC protocol**
   - No formal declaration of what IPC methods features require
   - Impact: Hidden dependencies between packages

### Low

1. **Context metrics location**
   - Currently in daemon, could be in core or feature-statusline
   - Not blocking but creates unclear ownership

## Missing Abstractions

| Abstraction | Purpose | Current State |
|-------------|---------|---------------|
| StateWriter interface | Contract for persistent state writes | 40+ ad-hoc implementations |
| Feature IPC Protocol | Formal declaration of IPC methods per feature | Embedded in daemon switch statement |
| PathResolver service | Centralized path building | Every package constructs paths independently |
| Feature configuration provider | Features declare config needs | Hardcoded values scattered |

## Recommendations for 9.5 (Feature Domain Consolidation)

### Phase 1: Formalize Contracts
1. Create `StateWriter` interface in @sidekick/types
2. Define Feature IPC Protocol with formal method declarations
3. Create `PathResolver` service in @sidekick/core

### Phase 2: Move Feature Code
1. Migrate P&R baseline logic (daemon.ts:624) to feature-reminders handler
2. Migrate VC state management to feature-reminders handler
3. Move reminder name constants to feature-reminders

### Phase 3: Clean Up Couplings
1. Consolidate file names as constants in @sidekick/types
2. Extract IPC method registry with validation
3. Move hardcoded timeouts to configuration

### Priority Fixes for 9.3
1. **FIXME at daemon.ts:624** - Move P&R baseline handling (explicit acknowledgment in code)
2. **Create PathResolver** - Highest ROI, eliminates 90+ duplications
3. **StateWriter interface** - Foundation for 9.3 State Management Infrastructure

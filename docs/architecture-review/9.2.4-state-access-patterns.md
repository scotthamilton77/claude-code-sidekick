# 9.2.4 State Access Pattern Inventory

## Summary
- Total state access points: 40+ distinct read/write locations
- Packages with state access: sidekick-daemon, sidekick-core, feature-statusline, feature-session-summary, feature-reminders, sidekick-ui, shared-providers
- Key concerns:
  - Path construction duplication across 7+ files
  - Inconsistent validation approaches (Zod vs try/catch)
  - Scattered raw fs calls instead of centralized service
  - Non-atomic writes in feature handlers

## Path Construction Patterns

### Pattern 1: Inline construction (most common, problematic)
```typescript
const summaryPath = path.join(stateDir, 'sessions', sessionId, 'state', 'session-summary.json');
```
Found in: handlers, readers, ui, tests (7+ locations for session-summary.json alone)

### Pattern 2: Via context helper
```typescript
const stateDir = context.paths.sessionStateDir(sessionId);
```
Found in: daemon handlers, some services

### Pattern 3: Via service methods
```typescript
stagingService.getStageDir(hook);
```
Found in: staging handlers (cleanest pattern)

**Issue**: Same paths constructed independently in multiple locations. Single point of change required if path structure evolves.

## Findings by Severity

### High (architectural issue)

| Finding | Files Affected | Impact |
|---------|----------------|--------|
| No centralized StateWriter abstraction | 40+ write locations | Inconsistent atomic writes, error handling |
| Path construction duplication | 7+ files for session-summary.json | Maintenance burden, potential drift |
| Missing validation on critical reads | TranscriptService, UI handlers | Silent corruption, runtime errors |
| Non-atomic writes in handlers | feature-session-summary handlers | Corruption risk on crash |

### Medium (inconsistency/duplication)

| Finding | Files Affected | Impact |
|---------|----------------|--------|
| Zod validation inconsistent | state-reader vs others | Some reads validated, some not |
| Separate log metrics aggregation logic | daemon.ts, state-reader.ts | Two sources of truth |
| Inconsistent error handling | state-reader, TranscriptService, handlers | Unclear "missing = OK" vs "missing = error" contract |
| Staleness detection only for metrics | TranscriptService only | UI can't detect stale non-metric state |

### Low (minor improvement)

| Finding | Files Affected | Impact |
|---------|----------------|--------|
| Magic string filenames | Multiple | 'session-summary.json' defined in 3+ places |
| No filesystem caching | StateReader | Repeated statusline renders = repeated I/O |
| Compaction history unbounded | TranscriptService | Grows indefinitely |
| Summary countdown has no schema | update-summary.ts | Silent corruption if file invalid |

## Recommendations for 9.3 (State Management Infrastructure)

1. **Create centralized StateWriter service** with:
   - Atomic write guarantee (tmp + rename)
   - Zod validation on read
   - Corrupt file recovery
   - Consistent error handling contract

2. **Centralize path construction** in `state-paths.ts`:
   - Named accessors for all state files
   - Single source of truth for path patterns
   - Type-safe session ID handling

3. **Add missing Zod schemas**:
   - summary-countdown.json
   - compaction-history.json

4. **Migrate all writes** to use StateWriter pattern

5. **Enhance StateReadResult** with:
   - Staleness indicators for UI polling
   - Clear "missing = default" vs "missing = error" semantics

6. **Consolidate log metrics** to single aggregation source

7. **Add compaction history pruning** (keep last N entries)

8. **Document state file lifecycle** (creation, update frequency, retention)

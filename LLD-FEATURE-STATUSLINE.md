# Feature: Statusline

NOTE: NOT YET REVIEWED

## 1. Overview

The Statusline feature provides a real-time, context-aware information bar to the user, typically displayed as part of the shell prompt or a dedicated status area. It aggregates data from the current session, the project state, and background analysis (Supervisor) to provide immediate feedback on costs, token usage, and session intent.

## 2. Scope & Responsibilities

### 2.1 Scope

- **Rendering**: Composes a formatted string based on a configurable template.
- **Data Aggregation**: Reads state from shared state files (written by Supervisor) and performs lightweight real-time checks (e.g., git branch).
- **Performance**: Must execute and render in **<50ms** to avoid perceptible lag in the shell prompt.

### 2.2 Responsibilities

1.  **Read Shared State**: Consume `session-state.json` and `session-summary.json` generated by the Supervisor.
2.  **Context Awareness**: Detect current git branch and working directory.
3.  **Formatting**: Apply color coding based on thresholds (e.g., token usage warnings) and user-defined themes.
4.  **Graceful Degradation**: Render partial information if state files are stale or missing, without crashing.

## 3. Data Architecture

The Statusline is a **reader-only** consumer of state. It does not write to state files or trigger heavy computations.

### 3.1 Input Data Sources

| Source                     | Data Points                                  | Access Method               |
| :------------------------- | :------------------------------------------- | :-------------------------- |
| **`session-state.json`**   | Token count, Cost, Duration, Model Name      | Read File (JSON)            |
| **`session-summary.json`** | Session Title, Latest Intent, Snarky Comment | Read File (JSON)            |
| **Git (Subprocess)**       | Current Branch                               | `git branch --show-current` |
| **Environment**            | CWD, Project Root                            | `process.cwd()`, Config     |

### 3.2 State File Optimization

_Critical Change from Legacy_: The legacy bash implementation calculated token counts by reading the entire transcript file on every render. In the Node.js architecture, this is moved to the **Supervisor**.

- **Supervisor** watches the transcript and updates `session-state.json` with the current token count and cost.
- **Statusline** reads `session-state.json` (small file), ensuring O(1) performance regardless of transcript size.

## 4. Configuration Schema

The feature is configured via the standard `sidekick` configuration system.

```typescript
interface StatuslineConfig {
  // Enable/Disable the feature
  enabled: boolean

  // The visual template string
  // Supported tokens: {model}, {tokens}, {cost}, {duration}, {branch}, {cwd}, {summary}
  format: string // Default: "[{model}] | {tokens} | {cwd}{branch} | {summary}"

  // Color thresholds
  thresholds: {
    tokens: {
      warning: number // Default: 100000
      critical: number // Default: 160000
    }
    cost: {
      warning: number // Default: 0.50
      critical: number // Default: 1.00
    }
  }

  // Visual preferences
  theme: {
    use_nerd_fonts: boolean // Default: true (uses icons like 🪙, ⎇)
    colors: {
      model: string // "blue"
      tokens: string // "green" (dynamic based on threshold)
      summary: string // "magenta"
    }
  }
}
```

## 5. Component Design

### 5.1 `StatuslineService`

The main entry point, responsible for orchestration.

```typescript
class StatuslineService {
  constructor(
    private config: ConfigService,
    private stateReader: StateReader,
    private gitProvider: GitProvider,
    private formatter: Formatter
  ) {}

  async render(): Promise<string> {
    // 1. Parallel data fetch
    const [state, summary, branch] = await Promise.all([
      this.stateReader.getSessionState(),
      this.stateReader.getSessionSummary(),
      this.gitProvider.getCurrentBranch(),
    ])

    // 2. Prepare view model (apply thresholds/logic)
    const viewModel = this.prepareViewModel(state, summary, branch)

    // 3. Render template
    return this.formatter.format(this.config.get('statusline.format'), viewModel)
  }
}
```

### 5.2 `StateReader`

Handles safe file reading with fallback defaults.

- Uses `fs.readFile` with a short timeout.
- Returns "empty" objects if files are missing or corrupt (JSON parse error).

### 5.3 `Formatter`

A lightweight string interpolator.

- Replaces `{token}` with values.
- Applies ANSI color codes based on the `theme` config.
- Handles conditional formatting (e.g., if `branch` is empty, don't show the separator).

## 6. Detailed Logic

### 6.1 Token & Cost Coloring

- **Green**: < Warning Threshold
- **Yellow**: >= Warning Threshold && < Critical Threshold
- **Red**: >= Critical Threshold

### 6.2 Summary Selection

Logic to decide what text to show in the `{summary}` slot:

1.  If `snarky_comment` exists -> Show `snarky_comment`.
2.  Else if `latest_intent` exists -> Show `latest_intent`.
3.  Else -> Show nothing or Session Title.

### 6.3 Git Integration

- Run `git branch --show-current` with a **10ms timeout**.
- If it times out or fails (not a git repo), return empty string.
- This prevents the statusline from hanging the shell in slow/networked filesystems.

## 7. CLI Command

The feature is exposed via the CLI:

```bash
sidekick statusline [--format <json|text>]
```

- **Default**: Outputs the rendered ANSI string.
- **`--format json`**: Outputs the raw data structure (useful for debugging or custom external renderers).

## 8. Resolution of Open Questions

### 8.1 Template Engine

**Decision**: Use a **custom lightweight interpolator**.

- **Rationale**: Full template engines (Handlebars/EJS) are too heavy for the <50ms requirement.
- **Implementation**: Regex-based replacement of `{key}`. Support simple conditional blocks if needed (e.g., `[?branch ...]`), but keep it simple for V1.

### 8.2 Stale State Handling

**Decision**: **Graceful Fallback**.

- If `session-state.json` is older than X seconds (e.g., 60s), the Supervisor might be down.
- **Action**: Render the stale data but apply a "dim" style or append a specific indicator (e.g., `(stale)`) to alert the user without breaking the flow.

### 8.3 Hook Contract

**Decision**: The hook invokes `sidekick statusline`.

- The shell script wrapper captures the output and displays it.
- No complex schema negotiation needed; the CLI simply outputs the string to stdout.

### 8.4 Refresh Cadence

**Decision**: **Pull-based (On-Demand)**.

- The statusline is re-rendered only when the shell prompts (via `PROMPT_COMMAND` or similar).
- It does **not** poll in the background.
- The **Supervisor** updates the state files in the background (Push model to state files), ensuring that when the Statusline reads them, they are reasonably fresh.
